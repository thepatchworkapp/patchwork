# Patchwork_iOS Agent Guide

This guide defines the operating rules for coding agents and contributors working in the
`Patchwork_iOS` project.

## Agent workflow

- For independent parallel work, use subagents to split the task into non-overlapping areas.
- For cross-cutting changes that touch multiple UI or feature areas, prefer one coordinating agent plus narrowly scoped worker tasks.
- Prefer the `Build iOS Apps` plugin capabilities for simulator builds, runs, tests, screenshots, and debugging when available.
- Use `xcodebuildmcp` tooling for simulator workflows and screenshots instead of ad hoc local scripting.

## Scope

- Scope: native iOS client in `Patchwork_iOS` only.
- Do not alter backend services; this app is a thin client over existing Convex/Better Auth contracts.
- Use this file as the primary source of truth for app-specific decisions, especially when release and ASC commands are involved.

## Mandatory instruction

- Do not code fallbacks or legacy support. This is a greenfield project. If the main code path does not work, it should be fixed. DO NOT CODE FALLBACKS.
- Fix the primary implementation path end-to-end rather than adding alternate branches for older OSes, compatibility layers, feature flags, or degraded behavior.
- Use available platform tools for workflows:
  - Use the `revenuecat` MCP server for RevenueCat paywall generation and project/configuration operations.
  - Use `xcodebuildmcp` project-aware tooling for simulator, build, archive, screenshot, and test operations.
  - Prefer the shared project tooling path over ad hoc shell scripts or raw `xcodebuild` command strings when both are available.
- Use the `asc` CLI for all App Store Connect operations (build distribution, testflight orchestration, release flow, and submission health).

- UI is SwiftUI with `NavigationStack` + `TabView`.
- Core domain state is held in:
  - `Patchwork/Core/SessionStore.swift` (session/auth token lifecycle)
  - `Patchwork/Core/AppState.swift` (app-level flow state)
- Network layer:
  - `Patchwork/Core/ConvexHTTPClient.swift` handles Convex API calls and Better Auth token exchange.
- Do not add local persistence layers unless explicitly scoped.

## Backend contracts (must follow)

- Convex endpoint patterns:
  - Queries: `POST /api/query` with `{ "path": "<namespace:function>", "args": { ... }, "format": "json" }`
  - Mutations: `POST /api/mutation` with same envelope.
- The app currently depends on these functions:
  - `categories:listCategories`
  - `conversations:getConversation`
  - `conversations:listConversations`
  - `jobs:completeJob`
  - `jobs:getJob`
  - `jobs:listJobs`
  - `messages:listMessages`
  - `messages:sendMessage`
  - `proposals:acceptProposal`
  - `proposals:counterProposal`
  - `proposals:declineProposal`
  - `proposals:sendProposal`
  - `reviews:canReview`
  - `search:searchTaskers`
  - `taskers:cancelSubscription`
  - `taskers:createTaskerProfile`
  - `taskers:getTaskerById`
  - `taskers:getTaskerProfile`
  - `taskers:setGhostMode`
  - `taskers:updateSubscriptionPlan`
  - `conversations:startConversation`
  - `users:createProfile`
  - `users:getCurrentUser`
  - `users:updateLocation`
- For filters and lists (role/status/category), send query args to backend. Never fetch broad results and filter client-side if a backend filter exists.
- Categories must come from `categories:listCategories`; do not hardcode category lists.
- Preserve and forward Better Auth cookie/auth fields exactly as current flow requires.

## Auth/OTP specifics (high risk area)

- OTP paths used by client:
  - `/api/auth/email-otp/send-verification-otp`
  - `/api/auth/sign-in/email-otp`
  - `/api/auth/convex/token`
- Observed historical failures to avoid:
  - `Unexpected response`
  - missing or null `Origin`
  - `Invalid origin`
- Required handling:
  - Preserve and forward Better Auth cookie values (`Set-Better-Auth-Cookie` / `Better-Auth-Cookie`).
  - Keep explicit `Origin` behavior stable.
  - Keep explicit fallback origin behavior for token retrieval as implemented.
  - Do not rely on shared cookie state leakage between auth attempts.
- Keep auth error handling direct and explicit. If token exchange fails for valid scenarios, fix request/headers and flow.

## UI and accessibility requirements

- Match the current PoC and Figma intent.
- Do not invent screens/actions outside product scope.
- Keep key controls and auth flow elements accessible:
  - `accessibilityIdentifier` for key inputs/buttons where useful
  - meaningful labels for VoiceOver
- Maintain existing layout intent and navigation behavior.
- For shared typography tokens, prefer Dynamic Type-friendly text styles over fixed font sizes.
- For `TabView`, prefer the modern `Tab(..., value:)` API over legacy `tabItem()` wiring when it compiles cleanly against the project target.
- Do not use `Text` concatenation with `+`; use interpolation or a dedicated view instead.
- Do not create `Binding(get:set:)` inline in view bodies when a bindable state path or extracted subview can own the binding directly.
- Icon-only controls still need real text labels; use `Label` and hide the title visually with `.labelStyle(.iconOnly)` when appropriate.
- Keep shared SwiftUI primitives split into focused files rather than growing catch-all component files.
- Use shared presentation primitives for polish work: prefer `PatchworkInlineStatusBanner` over one-off raw error text, and use the shared sheet chrome helper for modal styling rather than per-screen presentation tweaks.

## Build and run workflow

- Preferred local loop:
  1. Open `Patchwork.xcodeproj` in Xcode
  2. Select scheme `Patchwork`
  3. Run on simulator
- If using CLI:
  - build scheme `Patchwork`
  - use `xcodebuildmcp` or equivalent project-aware tooling instead of one-off simulator commands
  - run tests for both unit + UI as part of release-adjacent changes
- Use a current local simulator identifier for the target device family.

## Test policy

- Run unit tests and UI tests after significant logic or navigation changes.
- Ensure failures are root-caused by fixing the broken path; do not mask regressions with alternate behavior.
- Run iOS UI tests serially. Do not overlap `xcodebuild` UI test sessions against the same simulator or `DerivedData`.
- When running UI tests from CLI, always provide a unique `-resultBundlePath` for each run to avoid result-bundle corruption and runner bootstrap failures.
- Treat helper-backed UI scenarios as non-production-only unless the target deployment explicitly enables `/test-proxy`. Production disables testing helpers by design.

## Release/ASC expectations

- Confirm these before finalizing release work:
  - app slug and ID match the intended ASC target (`6759272540`)
  - build is processing and visible
  - export compliance is accepted
  - intended testers are present in assignment group
- If `MISSING_EXPORT_COMPLIANCE`, create/assign App Encryption Declaration in ASC.
- Keep `ITSAppUsesNonExemptEncryption = NO` in `Info.plist` unless policy changes.
- If upload reports duplicate bundle version, increment `CURRENT_PROJECT_VERSION`.
- Current release metadata source in source-of-truth config:
  - `MARKETING_VERSION=3.2`
  - `CURRENT_PROJECT_VERSION=29`

## Info.plist / packaging specifics

- Keep icon pipeline stable:
  - `CFBundleIconName = AppIcon`
  - `ASSETCATALOG_COMPILER_APPICON_NAME = AppIcon`
- Encryption key should remain `ITSAppUsesNonExemptEncryption = false` for ASC exemption workflow.
- Location usage text is required:
  - `NSLocationWhenInUseUsageDescription` is defined in Info.plist and should remain aligned with feature behavior.
- UI style is light-mode by configuration (`UIUserInterfaceStyle = Light`).

## Hard constraints

- Do not change backend endpoints/contracts.
- Do not alter Auth/session semantics unless required by protocol.
- Do not invent fallback routes for missing data; ensure primary data path is correct.
- Do not keep obsolete app identifiers.
- Do not proceed with release steps if critical release checks above are incomplete.

## Final sanity checklist for any change

- Confirm no accidental references to old app bundle IDs or legacy app slugs.
- Confirm all feature data comes from Convex and backend filters.
- Confirm location and auth flows still succeed with explicit headers/cookies.
- Confirm tests are green or fix root cause rather than bypassing.
- Confirm release-critical values (`MARKETING_VERSION`, `CURRENT_PROJECT_VERSION`) in `project.yml` reflect intent.
- Confirm any new accessibility-critical controls expose meaningful labels and identifiers where the surrounding screen already uses them.
